# Grape SPEC

## 产品定义

Grape 是一个给个人开发者使用的 macOS 原生 Git 工作台。

它默认直接回到上次打开的仓库，把当前改动、提交、分支、同步、历史和冲突处理这几段日常高频流程做深，而不是做成一个功能很多、但每天不够顺手的“大面板”。

第一版目标是可用的内部版，不是公开测试版。重点是把个人开发者最常见、最容易卡住、最容易出错的几段 Git 流程做成一套稳定、直觉、原生的日常体验。

## 第一版目标

- 给每天都在写代码的个人开发者使用
- 一次专注一个仓库，不做多仓库总控台
- 默认回到上次工作现场，减少重复进入步骤
- 把日常提交流、分支流、同步流和冲突处理做深
- 保持原生、安静、顺手，不靠堆功能制造“专业感”

## 界面布局方案

### 启动方式

- 默认直接打开上次使用的仓库
- 如果没有最近仓库，或者上次仓库路径失效，则进入仓库入口页
- 仓库入口页只负责两件事：打开本地仓库、克隆远端仓库

### 整体骨架

主界面采用“顶栏 + 常驻三栏”的固定结构。

- 顶栏负责状态提示和最高频全局动作
- 左侧栏负责仓库与分支上下文
- 中间主区负责当前改动与提交
- 右侧栏负责历史与差异预览

整个界面的主次关系固定为：中间做事，左右辅助，顶栏补充。

### 顶栏

顶栏是一条常驻轻量控制带，不替代三栏主结构。

**顶栏状态信息**

- 当前仓库名
- 当前分支
- 同步状态

**顶栏高频操作**

- 打开仓库
- 拉取
- 推送
- 新建分支
- 提交

顶栏按钮以图标为主，其中“提交”单独做成更明显的强调按钮。顶栏的任务不是塞满功能，而是让用户不离开当前视图，也能完成最常用的全局动作。

### 左侧栏

左侧栏是仓库与分支上下文区域，不承担主要内容阅读。

**左侧栏内容**

- 当前仓库信息
- 当前分支
- 同步状态摘要
- 未提交改动数量
- 分支管理入口

**左侧栏负责的动作**

- 新建分支
- 切换分支
- 重命名分支
- 删除分支
- 建立和管理远端追踪关系
- 整理分支

左侧栏的目标是让用户随时知道“我在哪个仓库、哪个分支、现在是什么状态”，而不是把内容挤成第二个主工作区。

### 中间主区

中间主区是最大区域，采用“上改动、下提交”的固定垂直流程。

**上半部分：当前改动**

- 改动文件列表
- 改动筛选
- 文件级选择提交
- 代码块级选择提交
- 丢弃与恢复改动入口
- 暂存区视图

**下半部分：提交区域**

- 提交说明输入
- 提交前状态确认
- 提交按钮

中间主区的目标是让“查看改动 -> 选择内容 -> 整理提交 -> 完成提交”成为一条连续动作，而不是在多个视图之间跳来跳去。

### 右侧栏

右侧栏采用“上历史、下预览”的固定上下结构。

**上半部分：历史**

- 提交关系图
- 最近提交列表
- 当前提交详情

**下半部分：预览**

- 差异预览
- 选中提交的详情预览

右侧栏的目标是让用户在处理当前改动时，仍然能快速回看最近发生了什么，不需要离开主工作流。

## 核心功能点清单

### 仓库入口

- 打开本地仓库
- 克隆远端仓库
- 返回最近仓库
- 自动回到上次打开的仓库

### 当前改动与提交

- 查看当前改动文件
- 查看具体差异
- 按文件选择提交内容
- 按代码块选择提交内容
- 暂存区视图
- 编写提交说明
- 提交前确认当前状态
- 完成提交
- 丢弃改动
- 恢复改动

### 分支管理

- 新建分支
- 切换分支
- 重命名分支
- 删除分支
- 建立远端追踪关系
- 查看远端追踪关系
- 整理分支
- 合并分支

### 同步

- 拉取
- 推送
- 查看同步结果
- 查看本地领先或落后远端的状态

### 历史与预览

- 查看提交关系图
- 查看最近历史
- 查看提交详情
- 查看差异预览

### 冲突与保护

- 完整三方冲突处理界面
- 冲突文件列表
- 冲突解决状态提示
- 风险操作确认
- 在危险动作前明确说明后果

## 关键流程与状态

### 启动进入

- 有可用最近仓库时，直接进入上次仓库
- 没有最近仓库时，进入仓库入口页
- 上次仓库路径失效时，清楚提示仓库不可用，并回到仓库入口页

### 正常提交流

- 用户进入仓库后先看到当前改动
- 用户选择文件或代码块
- 用户在提交区填写说明
- 满足条件后完成提交
- 提交完成后，界面明确反馈结果，并更新历史与同步状态

### 部分提交流

- 用户可以只选择部分文件
- 用户也可以在单个文件内只选择部分代码块
- 未被选择的内容继续保留在当前工作区
- 暂存区视图需要清楚区分“准备提交的”和“未提交的”

### 丢弃与恢复改动

- 用户可以对单个文件或部分改动执行丢弃
- 用户触发前必须看到明确后果说明
- 恢复动作必须让用户知道会回到哪个状态
- 危险操作不允许无提示直接执行

### 切换分支遇到未处理改动

- 如果切换分支会影响当前改动，先提示风险
- 提示中要直接说明可能发生什么
- 用户可以取消切换，或先返回处理中间主区的改动

### 拉取与推送

- 拉取成功后明确告诉用户是否有新内容进入本地
- 推送成功后明确告诉用户远端已更新
- 失败时直接说明失败类型，而不是只给抽象错误

### 冲突处理

- 拉取、推送、合并后如果出现冲突，直接进入应用内三方冲突处理界面
- 用户能看到哪些文件冲突
- 用户能看到每个冲突文件的处理进度
- 冲突解决完成后，界面明确给出下一步

### 关键状态

**正常状态**

- 没有改动
- 有改动但尚未选择任何内容
- 已有选择内容但提交说明为空
- 可正常提交
- 本地领先远端
- 本地落后远端
- 本地与远端一致

**异常状态**

- 上次仓库路径失效
- 拉取失败
- 推送失败
- 认证失败
- 仓库状态异常
- 存在冲突

所有状态都要说人话，让用户知道现在发生了什么、下一步该做什么。

## 第一版不做什么

第一版明确不包含这些内容：

- GitHub
- Pull Request
- Issue
- 多仓库总控台
- 团队协作面板
- 新建仓库
- stash / shelve
- 大范围搜索筛选
- 面向少数熟练用户的高级 Git 工具箱

第一版不追求做全，也不追求把所有 Git 命令都图形化。它只做个人开发者每天真的会反复碰到的高频主线。

## 设计原则

- 原生、安静、顺手
- 顶栏轻，三栏稳
- 中间做事，左右辅助
- 结果说人话
- 危险动作先说后果
- 优先做深高频，不扩张平台能力

- Grape 给谁用
- 启动后先看到什么
- 顶栏放什么
- 三栏分别负责什么
- 第一版核心功能有哪些
- 哪些内容明确不做
- 提交筛选做到什么颗粒度
- 是否有暂存区视图
- 是否支持克隆远端仓库
- 是否内置三方冲突处理
- 是否支持丢弃与恢复改动

如果这些问题里有任何一个还需要靠猜，这份 SPEC 就还不够完整。
