## 执行摘要

Grape 是一款面向个人开发者的 macOS 原生 Git 日常工作台，目标不是做成“大而全”的 Git 平台，而是把用户每天最常发生、最容易卡住、最容易出错的几段 Git 主流程做成一套安静、顺手、可信任的桌面体验。

这份 `PRD.md` 负责定义 Grape 的需求层判断，包括：为什么要做、服务谁、解决什么问题、第一版承诺什么、如何衡量产品是否成立、哪些能力暂时明确不做，以及后续路线如何演进。页面级结构、流程状态和交互细节由 `SPEC.md` 继续承接，技术实现与依赖选择由 `TECH.md` 继续承接。

当前产品策略分为两层：

- `v1`：承诺交付一条完整可用的单仓库日常 Git 主路径，覆盖仓库进入、查看改动、部分提交、分支切换、同步远端、查看最近历史和冲突处理。
- 长期蓝图：在不破坏当前定位的前提下，逐步把历史、同步、冲突处理与更大管理视角打磨到更成熟，但不在当前版本承诺 GitHub、Pull Request、多仓库总控或团队协作平台能力。

## 背景与机会

### 背景

对个人开发者来说，Git 是每天都会发生的基础操作，但现有图形工具常见地落入三种问题：

- 功能堆叠过重，界面密度高，用户需要先适应工具而不是直接开始工作。
- 危险动作保护不足，用户在切换分支、丢弃改动、同步远端时缺少足够清楚的后果提示。
- 产品要么太像团队协作平台的附属面板，要么只是把命令行重新包装一层，没有形成真正顺手的日常桌面体验。

个人开发者的真实高频需求并不复杂，但要求极高：

- 要能快速进入正在处理的仓库和当前工作现场。
- 要能清楚地看见改动，并放心地挑选哪些内容进入一次提交。
- 要能在分支、同步、历史与冲突之间连续完成任务，而不是频繁跳回终端。
- 要在危险操作前得到清楚、可信、说人话的反馈。

### 机会

Grape 的机会不在于功能数量，而在于体验密度和信任感：

- 为个人开发者提供真正原生、安静、顺手的 macOS Git 工具。
- 把“看改动 -> 选内容 -> 提交 -> 切分支 -> 同步 -> 看历史”做成一条连续主路径。
- 在危险场景下建立强保护机制，让用户更愿意在 GUI 内完成操作，而不是退回终端求稳。
- 用更轻、更清晰的产品定位，与偏团队、偏平台、偏效率中心的现有工具形成差异。

## 目标用户与 JTBD

### 核心目标用户

#### 1. 个人开发者

特征：

- 每天都在写代码
- 主要维护自己的项目或少量仓库
- 对 Git 有基本理解，但不希望每次都切回终端
- 关注效率，也关注安全感和可解释性

核心任务：

- 当我正在开发一个功能时，我想快速恢复上次仓库现场，这样我可以立即继续工作。
- 当我准备提交时，我想看清改动并选择正确的文件或代码块，这样我可以写出干净、可控的提交。
- 当我切换分支或同步远端时，我想先知道会发生什么，这样我不会莫名其妙丢改动或搞乱当前状态。

#### 2. 维护少量仓库的独立开发者

特征：

- 同时维护少量个人项目、实验项目或客户端项目
- 需要更可靠的历史查看、同步反馈和冲突处理
- 不需要团队协作面板，但需要稳定、可信任的日常 Git 操作体验

核心任务：

- 当我在多个个人项目间轮换时，我想快速打开或恢复最近仓库，这样我不需要重复找路径和重建上下文。
- 当同步失败或出现冲突时，我想得到明确下一步，这样我可以在应用内解决问题而不是自己拼命推断。

### 非目标用户

- 以 GitHub / Pull Request / Issue 为主要工作入口的团队协作用户
- 管理大量仓库并需要统一全局视角的高级多仓库用户
- 依赖复杂企业级 DevOps 流程、权限流和平台集成的组织型用户
- 主要追求高级低频 Git 工具箱能力的熟练极客用户

## 产品定位与价值主张

### 产品定位

Grape 是一款面向个人开发者的 macOS 原生 Git 日常工作台，专注单仓库高频主线，不追求平台化堆叠，不试图覆盖全部 Git 场景。

### 核心价值主张

- 原生：像真正的 Mac 应用，而不是网页壳或复杂控制台。
- 安静：把高频动作放到前面，不用多余功能打断思路。
- 顺手：让主任务连续发生，减少来回跳转和上下文切换。
- 可信：危险动作有保护，失败反馈说人话，用户知道发生了什么和下一步该做什么。

### 差异化判断

Grape 不想成为：

- 一个大而全的 Git 效率中心
- 一个以团队协作平台为中心的附属客户端
- 一个只给熟练用户准备的密集工具箱

Grape 想成为：

- 一款你愿意每天开着的 Mac 原生 Git 工具
- 一款能让个人开发者在不打开终端的情况下走通主流程的桌面应用

## 产品目标

### v1 结果目标

1. 让个人开发者在不打开终端的情况下，完成一次完整的日常 Git 主路径。
2. 降低日常 Git 操作的认知负担，让用户更容易看懂当前状态、风险与下一步。
3. 提升高频流程连续性，把查看改动、提交、切换分支、同步和查看历史串成一条自然路径。
4. 强化危险操作保护，降低误丢改动、误切分支和错误同步带来的焦虑。
5. 建立 Grape 作为“原生、安静、顺手”的 macOS Git 工具心智。

### 长期结果目标

1. 在保持单仓库体验优势的前提下，逐步扩大历史、冲突和同步处理深度。
2. 为未来更大的管理视角或平台能力留下空间，但不牺牲当前产品定位。
3. 把 Grape 从“能完成任务”打磨成“愿意长期常开”的日常工具。

## 非目标

以下内容不属于 v1 承诺范围：

- GitHub 集成
- Pull Request 管理
- Issue 管理
- 多仓库总控台
- 团队协作面板
- 新建仓库
- stash / shelve
- 面向少数熟练用户的高级低频 Git 工具箱
- 复杂企业级权限、策略、平台集成能力

以下内容属于长期方向，但不构成当前版本承诺：

- 多仓库统一概览
- 团队协作上下文
- 外部平台整合
- 更广泛的高级 Git 工作流支持

## 核心场景与主路径

### 场景 1：打开或恢复仓库

- 触发条件：用户启动应用、打开本地仓库、克隆远端仓库，或回到最近项目
- 用户意图：尽快回到一个可继续工作的仓库上下文
- 成功结果：用户无需手动重新定位，即可进入当前仓库、看到当前状态并继续操作
- 失败反馈要求：如果仓库路径失效、仓库异常或克隆失败，应用必须直接说明原因与下一步

### 场景 2：查看改动并部分提交

- 触发条件：用户完成一段开发，需要整理本次改动
- 用户意图：明确区分哪些内容应该进入当前提交，哪些内容应该继续留在工作区
- 成功结果：用户看清差异，完成文件级或代码块级选择，写好提交说明，并得到明确提交结果
- 失败反馈要求：提交失败时应用必须说明失败原因、受影响状态和建议操作

### 场景 3：切换分支与同步远端

- 触发条件：用户需要切换任务、拉取最新内容或推送本地更新
- 用户意图：在不破坏当前工作状态的前提下，安全完成分支与同步操作
- 成功结果：用户知道当前分支、本地与远端关系、同步结果和后续状态
- 失败反馈要求：如果当前改动阻止切换、拉取失败、推送失败或认证失败，应用必须给出具体后果说明和下一步建议

### 场景 4：冲突出现后的进入与退出处理

- 触发条件：拉取、推送、合并或切换导致冲突
- 用户意图：看清哪些文件冲突、当前进度是什么、应该如何继续
- 成功结果：用户在应用内完成冲突处理，看到处理完成后的明确下一步
- 失败反馈要求：应用不能只展示抽象错误，必须让用户知道冲突来自哪里、还剩什么未完成

### 核心主路径

Grape v1 的核心主路径定义为：

打开仓库或恢复最近仓库 -> 查看当前改动 -> 选择提交内容 -> 编写提交说明并提交 -> 切换分支或同步远端 -> 查看最近历史

这个主路径必须连续、清楚、低噪音，并在关键节点提供风险提示和反馈。

### 假设

- 目标用户已经具备基本 Git 概念，不需要产品承担 Git 教学工具角色。
- 用户愿意为了更清晰、更安全的主路径放弃一部分高级低频 Git 能力。
- 指标统计能力可在实现阶段补齐，不阻碍当前 PRD 先锁定目标值和验收口径。
- 对外平台整合不是当前产品成立的前提。

## 发布与验收标准

### 文档验收

`PRD.md` 完成后，读者应能明确回答：

- Grape 为什么值得做
- Grape 优先服务谁，不服务谁
- Grape v1 承诺做什么，不做什么
- Grape 依靠哪些核心场景建立价值
- Grape 如何被衡量为“成功”
- Grape 的路线图如何区分 Now / Next / Later

如果上述问题仍需要依赖猜测，说明 PRD 还不够完整。

### 产品验收

产品进入 v1 验收时，至少应满足：

- 用户能在不打开终端的情况下完成核心主路径
- 仓库入口、当前改动、部分提交、分支切换、同步、最近历史和冲突处理均形成闭环
- 所有危险操作均有明确确认与后果说明
- 常见失败场景均能提供可操作反馈
- 明确 out-of-scope 的能力没有被误写成已承诺交付项
- 成功指标具备可验证口径

## 关联文档与边界说明

### 文档职责

- `README.md`
  - 负责对外介绍、产品愿景、品牌方向与简要范围说明。
- `SPEC.md`
  - 负责页面结构、功能行为、流程状态、异常状态和交互层规格。
- `TECH.md`
  - 负责技术选型、架构分层、数据结构、Git 调用和实现边界。
- `PRD.md`
  - 负责用户、问题、需求、目标、指标、范围、优先级、路线图和验收口径。

### PRD 明确不负责的内容

- 不重复 Git 底层命令、数据结构或架构实现细节
- 不展开页面级交互规格和所有状态流转细节
- 不替代 `SPEC.md` 的界面与行为定义
- 不替代 `TECH.md` 的技术落地方案

### 文档协同原则

- 需求变化先体现在 `PRD.md`
- 行为与界面变化体现在 `SPEC.md`
- 技术方向与实现边界变化体现在 `TECH.md`
- 三份文档发生冲突时，应先回到产品需求判断，再同步更新相关文档，而不是长期允许文档分叉
